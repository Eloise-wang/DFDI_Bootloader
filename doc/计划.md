# DFDI_Bootloader 完整功能实现计划

## 1. 当前状态评估

当前工程已经具备 Bootloader 主体框架，主要链路已经串通：

- `BOOTLOADER_MAIN_Init()` 已接入 `TP_Init()`、`UDS_Init()`、`FLASH_APP_Init()`、`BSP_Flash_APPAddrCheck()`
- `Flash_OperateMainFunction()` 已进入主循环
- `unifiedStack_fls_app.c` 已改为通过 `BSP_Flash_*` 接口访问 Flash
- `bsp_flash.c` 已存在并实现了基础 Flash 适配层
- `flash_hal.c` 已从当前工程编译链中移除

当前工作重点已经不是协议框架搭建，而是把 Flash 适配层、Boot 跳转逻辑和下载闭环做成可验证、可量产的版本。

---

## 2. 总体目标

实现一个可稳定运行的 AC78406 Bootloader，满足以下能力：

1. 支持通过 UDS + CAN TP 完整下载应用
2. 支持 APP 区擦除、写入、校验、状态记录
3. 支持 Bootloader 对有效 APP 的正确判定与跳转
4. 支持异常中断场景下的安全保护，避免跳转到损坏 APP
5. 具备基础的 A/B 分区管理能力

---

## 3. 分阶段实施计划

### 阶段一：完善 Flash 适配层（最高优先级）

目标：把 `bsp_flash.c` 从“接口可用”提升到“真实可烧写、可保护、可验证”的版本。

#### 3.1 需要完善的函数

- `BSP_Flash_APPAddrCheck()`
- `BSP_Flash_Hal_EraseSector()`
- `BSP_Flash_Hal_ProgramData()`
- `BSP_Flash_WriteAPPInfoData()`
- `BSP_Flash_GetVectorTableOffset()`

#### 3.2 需要补充的检查与保护

1. 地址合法性检查
   - APP_A、APP_B 分区必须位于 PFlash 合法范围内
   - Bootloader 区与 APP 区不能重叠
   - APP_A 与 APP_B 不能重叠
   - APP 信息区必须落在设计范围内

2. 对齐检查
   - 擦除起始地址必须满足扇区对齐
   - 写入起始地址必须满足编程粒度对齐
   - 写入长度必须满足编程粒度要求，或定义补齐策略

3. 写保护
   - 禁止写 Bootloader 区
   - 禁止写非法保留区
   - 禁止越界写

4. 写后校验
   - Flash 编程完成后建议增加读回校验
   - APP 信息区写入后必须校验内容一致性

5. 返回值与错误路径统一
   - 每个 Flash 操作接口必须保证失败时可准确返回 `FALSE`
   - 避免初始化失败后仍继续执行写擦操作

#### 3.3 阶段一输出物

- 完整可靠的 `bsp_flash.c`
- 明确的 Flash 分区校验逻辑
- 可以支持 `unifiedStack_fls_app.c` 稳定调用的 Flash API

---

### 阶段二：梳理并固定 Boot 跳转 APP 逻辑

目标：确保下载成功后，Bootloader 能准确判断 APP 是否有效，并安全跳转。

#### 4.1 需要重点梳理的文件

- `boot.c`
- `boot_cfg.c`
- `unifiedStack_bootloader_main.c`
- 与 `tAppFlashStatus` 相关的存取逻辑

#### 4.2 需要明确的关键问题

1. Bootloader 判断 APP 有效的条件是什么
   - 是否需要 `isFlashErasedSuccessful`
   - 是否需要 `isFlashProgramSuccessful`
   - 是否需要 CRC 校验通过
   - 是否需要 APP 启动地址有效

2. Bootloader 最终跳转到哪里
   - APP 起始地址
   - ResetHandler 地址
   - `tAppFlashStatus` 中保存的启动地址

3. 跳转前是否完成必要准备
   - 关闭中断
   - 设置 MSP
   - 设置 VTOR
   - 清理必要状态

4. A/B APP 的启动优先级
   - 优先启动最新下载成功的 APP
   - 或优先启动 APP_A / APP_B
   - 失败后是否支持回退

#### 4.3 阶段二输出物

- 一套明确的 APP 有效性判定规则
- 一套明确的 Boot 跳转流程
- 必要时修正 `BSP_Flash_GetVectorTableOffset()` 和跳转代码

---

### 阶段三：打通 UDS 下载完整闭环

目标：使 Bootloader 支持标准下载流程，并完成 APP 擦写、数据接收、状态保存。

#### 5.1 重点模块

- `unifiedStack_fls_app.c`
- `unified_uds_app.c`
- `unifiedStack_uds_app_cfg.c`
- `unifiedStack_TP.c`
- `unifiedStack_can_tp.c`

#### 5.2 需要确认的 UDS 服务

至少需要验证以下服务：

- `0x10` Diagnostic Session Control
- `0x27` Security Access
- `0x34` Request Download
- `0x36` Transfer Data
- `0x37` Request Transfer Exit
- `0x31` Routine Control
- `0x11` ECU Reset
- 必要时 `0x3E` Tester Present

#### 5.3 需要重点检查的下载逻辑

1. Request Download
   - 是否正确解析起始地址与长度
   - 是否正确选择 APP_A / APP_B
   - 是否正确设置当前下载状态

2. Transfer Data
   - 是否支持连续多帧接收
   - 是否支持长度累加
   - 是否防止越界和长度溢出
   - 是否处理非 8 字节对齐末包

3. Transfer Exit
   - 是否完成剩余缓冲写入
   - 是否完成 CRC/完整性处理
   - 是否完成 APP 状态区更新

4. Routine Control
   - 擦除流程是否与下载长度一致
   - 校验流程是否与实际 Flash 内容一致

#### 5.4 阶段三输出物

- 可跑通的一次完整下载链路
- APP 数据正确落盘
- 下载结束后状态区正确更新

---

### 阶段四：增加异常保护与容错逻辑

目标：避免中途掉电、非法下载或错误数据导致设备无法启动。

#### 6.1 建议增加的保护项

1. 下载开始前
   - 将目标 APP 状态置为“无效/未完成”

2. 下载过程中
   - 记录当前写入进度
   - 严格限制下载范围
   - 异常返回时保留失败状态

3. 下载完成后
   - 只有在擦除、写入、CRC、启动地址都正确时，才将 APP 标记为有效

4. Boot 启动阶段
   - 如果 APP 状态不完整或 CRC 错误，则禁止跳转 APP
   - 保持在 Bootloader 等待重新下载

#### 6.2 可选增强项

- 支持下载失败回滚
- 支持 A/B 镜像回退策略
- 支持更细粒度错误码上报

---

### 阶段五：联调与实机验证

目标：确认系统在正常和异常场景下都能稳定工作。

#### 7.1 正常流程测试

1. 上电无下载请求，正常跳转 APP
2. 上电有下载请求，停留在 Bootloader
3. 完整执行一次 UDS 下载流程
4. 下载完成后复位，成功跳转新 APP

#### 7.2 边界条件测试

1. 非法下载地址
2. 下载长度超分区范围
3. 非对齐数据长度
4. TransferData 序号异常
5. CRC 错误

#### 7.3 异常场景测试

1. 擦除中断电
2. 写入中断电
3. 下载中途通信中断
4. APP 状态区损坏
5. APP ResetHandler 非法

#### 7.4 A/B 场景测试

1. 下载到 APP_A 并启动
2. 下载到 APP_B 并启动
3. 最新版本识别是否正确
4. 旧版本是否可作为保底镜像

---

## 4. 推荐实施顺序

建议严格按以下顺序推进：

1. **先完成阶段一：Flash 适配层完善**
2. **再完成阶段二：Boot 跳转逻辑梳理**
3. **然后完成阶段三：下载链路联调**
4. **最后做阶段四和阶段五：保护增强与实机验证**

这样做的原因：

- 协议层目前主体已经在
- 当前最大风险点在 Flash 可写性和启动闭环
- 如果先改 UDS 细节而 Flash 与跳转逻辑不稳，后续调试成本会非常高
